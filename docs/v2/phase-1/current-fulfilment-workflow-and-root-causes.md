# V2 Phase 1 — Current fulfilment workflow and confirmed root causes

Date: 15 July 2026

Base SHA: `6f5d6e110dad2bb818cedfebbf037abc962d29ab`
Scope: production-compatible manual fulfilment only. Migration `0017_phase14_canonical_disabled_foundation.sql` was not applied.

## Evidence-based workflow trace at the base SHA

1. Assessment creation and completion are handled under `/score/assessment/*`; the respondent writes answers and submits through the assessment API routes.
2. Deterministic scoring persists a completed, locked `score_runs` row and links it through `assessments.current_score_run_id`.
3. A detailed-report request calls `createOrGetOrderForReportRequest` in `src/lib/orders/manual-eft-orders.ts`. It creates an `orders` row in `awaiting_payment`, an `order_events` row and Phase 13 analytics/internal-notification records.
4. An authorised finance admin changes the order to `payment_received` through the order status route. Phase 14 auto-fulfilment is guarded by disabled flags, but the route still contained dormant Phase 14 workflow-start code.
5. The admin order page posts Generate Report to the generation route.
6. That route called `generatePremiumReport`, whose first permitted-admin step called `requirePhase14Action` and Phase 14 RPCs such as `authorize_phase14_action` and `ensure_manual_premium_report_fulfilment`.
7. Those RPCs and tables are defined by migration 0017, which is deliberately unapplied. The permitted manual action therefore depended on the disabled architecture.
8. The older deterministic rendering path still existed as static assembly, content-selection, HTML and Chromium helpers, but the current route no longer invoked it independently of Phase 14.
9. Download similarly called `requirePhase14Action`, `assert_premium_report_download_entitlement` and `record_phase14_report_download`; delivery called the Phase 14 delivery service.
10. The order UI queried `report_fulfilments` and silently returned `null` when that unavailable Phase 14 table failed. The order queue displayed payment state only.

## Defect reproduction and root-cause matrix

| Defect | Reproduction at base SHA | Observed behaviour | Confirmed root cause and evidence | Layer / affected state | Smallest safe remediation |
|---|---|---|---|---|---|
| DEF-008 | Create or reuse a detailed-report order without `MK_INTERNAL_LEADS_EMAIL`. | No reliable admin notification; no customer order-confirmation email event; a skipped internal notification was not visible on the order page. | `configuredRecipient()` returned `null`; `queueInternalNotification()` returned `skipped_no_recipient` unless strict mode was requested. `trackEftOrderEvent()` did not inspect the result. Only the admin/internal event existed; no customer confirmation event was created. | Backend/configuration; `email_events`, `assessment_events`; `manual-eft-orders.ts`, `internal-notifications.ts`. | Persist two idempotent disabled-provider events with explicit recipients and safe payloads, expose their states in admin UI, and retain no-send mode. |
| DEF-009 (inspection only) | Mark an eligible order `payment_received`. | Automatic generation was not expected while flags were disabled, but the status route retained Phase 14 queue/capability/workflow code. | `status/route.ts` conditionally imported and invoked `queuePremiumReportFulfilment`, Phase 14 worker authorisation and workflow start. | Backend/configuration; Phase 14 tables/RPCs if activated. | Remove automatic-generation code from the Phase 1 payment route. Payment confirmation remains a recorded manual transition. |
| DEF-010 | Sign in as a generation-capable admin and submit Generate Report while migration 0017 remains unapplied. | Generation fails before the deterministic path; the old page did not render the `report_error` query parameter, so an HTML submit appeared silent. | The route called `generatePremiumReport`; it called `requirePhase14Action` and 0017-only RPCs. The base page created redirect query parameters but did not read them. | Frontend/backend/database/auth; generation route, `premium-report-service-core.ts`, `phase14-security.ts`. | Route the action to a production-compatible deterministic manual service and render every result. |
| DEF-011 | Open a generated report and invoke Send Email under the permitted Phase 1 schema. | Delivery depended on Phase 14 authorisation/provider flow and had no explicit disabled/manual state. | `send-email/route.ts` called `deliverPremiumReportEmail`, which is a Phase 14 gated service. | Backend/database/configuration; `email_events`, Phase 14 delivery tables/RPCs. | Add a separate manual delivery-attempt model using only `disabled` or `double` provider mode. |
| DEF-012 | Download a report as an otherwise authorised admin with migration 0017 unapplied. | The action returned a broad `download_denied` response. | `premium-report-download.ts` required `authorize_phase14_action`, an 0017-only entitlement RPC and an 0017-only audit RPC. | DEF-013 | Open the order detail page after payment/generation failure. | Payment was visible, but generation, storage and delivery states were missing or derived from an unavailable Phase 14 table. | `getLatestFulfilment()` queried `report_fulfilments` and returned `null` on any query error. | DEF-014 | Open `/score/admin/orders` with paid unresolved orders. | The queue showed order payment status only; there was no paid/no-report or ready/not-delivered exception view. | `getAdminOrderList()` and the page did not load reports, generation or delivery state. | Frontend/backend/database. | Derive operational filters from persisted order/report/attempt state and keep all orders visible. |
| DEF-015 | Generate, preview, download or deliver a report, then inspect the order. | The order timeline contained order status events only; report events and email events were separate and not presented as a unified operational history. | `getAdminOrderDetail()` loaded only `order_events` and order audit rows; base generation/download did not add complete order-level transition events. | Frontend/backend/database. | Record each manual transition in `order_events` with safe references and render it with generation/delivery histories. |
| DEF-016 | Double-click Generate or submit concurrently from two admins. | The permitted pre-Phase-14 deterministic path had no active-attempt row or database uniqueness boundary; the current Phase 14 claim could not be reached safely while disabled. | The Phase 10 route calculated versions and uploaded without a manual active-attempt unique index. The current route delegated concurrency to unapplied 0017. | Backend/database/storage. | Add a request-key unique constraint and a partial unique index allowing one queued/generating attempt per order; disable the client button while active. |
| DEF-020 | Inspect a generated `reports` row and private storage object. | Bucket/path/checksum existed, but file name, MIME type, size, organisation relationship and verified storage state were not explicit. | The base `reports` schema had only bucket/path/checksum; the Phase 10 upload did not read the object back before marking the report generated. | Database/storage. | Add metadata columns, deterministic organisation/order/version paths, immutable upload and read-back checksum validation. |
| DEF-021 | Authenticate through the current admin cookie and invoke Download/Generate. | Base route auth succeeded, then a second Phase 14 AAL2/RPC authorisation boundary failed or was unavailable. | `getAdminSession()` used current user plus active profile; `requirePhase14Action()` separately required an authenticated client and `authorize_phase14_action`, which is absent without 0017 and enforces the dormant Phase 14 gate. | Auth/backend/database. | Use the existing active-admin role model for Phase 1 routes and database RPCs; do not weaken RLS or satisfy Phase 14 gates. |
| DEF-022 | Force render/storage/download/delivery failures and inspect the admin dashboard. | Console messages or broad responses existed, but no guaranteed attempt record, technical reference or exception queue was created. | Failure could occur before a report row existed; `safeLogReportAttempt()` could only create `report_events` when a report ID existed, and operational pages did not query failures. | Backend/database/frontend/observability. | Persist every attempt before work starts, transition it on failure, add safe structured logs and surface the persisted failure in queues/timeline. |

## Affected production-compatible structures

- Existing tables reused: `orders`, `order_events`, `assessments`, `score_runs`, `reports`, `report_events`, `email_events`, `audit_logs`, `report_templates`, `report_content_blocks`.
- Existing private bucket reused and reasserted: `generated-reports`.
- New Phase 1 tables: `manual_report_generation_attempts`, `manual_report_delivery_attempts`.
- New Phase 1 RPCs: claim/start/complete/fail manual generation and claim/complete manual delivery.
- New object path: `<organisation UUID>/<order UUID>/v<version>/<report reference>-<checksum prefix>.pdf`.

No cause above is labelled confirmed unless it is directly evidenced by the base source or the production-compatible schema replay.
